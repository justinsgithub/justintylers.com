There's an insane amount of valuable data sitting in public databases. State licensing boards, open data portals, government registries. It's all there, free to access. The problem is getting it out.

These aren't clean APIs with nice documentation. They're clunky government websites with inconsistent formats, anti-scraping measures, and data scattered across dozens of sources. Extracting it manually is mind-numbing. Building custom scrapers for each one takes real engineering effort.

That's where AI changes the equation.

## The Project

A client needed contractor licensing data from every state. Business names, license types, addresses, phone numbers. Not a hundred records. Hundreds of thousands.

The data exists because every state requires contractors to be licensed, and that licensing information is public record. But each state stores and presents it differently. Texas organizes by county. Florida by license category. New York City has three separate databases. North Carolina requires querying all 855 zip codes individually.

I ended up building scrapers for 65+ sources. Here's the process.

## Step 1: Source Discovery

Before writing any code, figure out where the data lives. For any domain of public data, there are usually three categories:

**Government portals.** The primary source. For contractor data, that means state licensing boards: Texas TDLR, Florida DBPR, California CSLB, Utah DOPL. These have the most complete data but often require navigating dated interfaces with server-side rendering and pagination.

**Open data portals.** Many cities and states publish data through Socrata or ArcGIS APIs. Dallas, Chicago, San Diego, Orlando, Seattle. These are the easiest to work with because the data is already structured and accessible through standardized API calls.

**Bulk downloads.** Some agencies publish downloadable CSVs or databases. California's contractor licensing board publishes a "License Master" CSV with nearly 100,000 records. Arkansas publishes their full list as a downloadable file. These are gold mines when they exist.

The lesson: always check for an API or bulk download before building a scraper. Scraping should be the last resort, not the first instinct.

## Step 2: Building the Scrapers

This is where Claude Code changes the game. Each source needs its own scraper because every website is different. Without AI, building 65+ scrapers would take months. With Claude Code, I could describe the target site structure and get a working first draft in minutes.

For JavaScript-rendered sites (most state portals), the scrapers use Selenium to drive a real browser. For static HTML, BeautifulSoup. For APIs, direct HTTP requests with pagination handling.

The key architectural decision: every scraper outputs the same JSONL format regardless of source. One JSON object per line, standardized field names. This means the entire downstream pipeline works the same way no matter where the data came from.

```
{"business_name": "Smith Electric LLC",
 "license_number": "EC-12345",
 "license_type": "Electrical Contractor",
 "license_status": "Active",
 "phone": "5551234567",
 "address": {"street": "123 Main St", "city": "Austin",
             "state": "TX", "zip": "78701"}}
```

Each scraper handles its own edge cases: pagination, rate limiting, session expiration, retry logic, and progress saving for resume capability. The progress saving matters. Large scrapes take hours. You don't want to start over because your internet hiccupped.

## Step 3: Handling Anti-Scraping Measures

Government sites aren't as aggressive as commercial platforms, but they still fight scrapers. Rate limiting is the most common defense. Hit a site too fast and you're blocked.

The countermeasures:
- **Rate limiting**: Deliberate delays between requests. Too fast and you're blocked. Too slow and the scrape takes weeks.
- **IP rotation**: VPN cycling to distribute requests across different IP addresses.
- **Session management**: Some sites expire sessions after a timeout or certain number of requests. The scraper needs to detect this and re-authenticate.
- **CAPTCHA handling**: Some sources require solving CAPTCHAs. In those cases, manual intervention or switching to a bulk download approach.

The balance is speed vs. detection. You want to scrape as fast as possible without triggering blocks. That threshold is different for every site.

## Step 4: Data Normalization

Raw scraped data is messy. Phone numbers come in every format imaginable: (555) 123-4567, 555-123-4567, 5551234567, +1 555 123 4567. Business names have inconsistent capitalization, trailing whitespace, and special characters. Addresses are sometimes one field, sometimes five.

The normalization step standardizes everything:
- Phone numbers stripped to 10 digits
- Business names trimmed and consistently cased
- Addresses parsed into structured components
- Status values mapped to standard categories
- Invalid or expired records filtered out

This is exactly the kind of work AI excels at. Describe the normalization rules, Claude writes the transforms, you verify the output on a sample and run it across everything.

## Step 5: Data Enrichment

Scraped data often has gaps. For this project, the biggest gap was phone numbers. State licensing records frequently include business name and address but no phone number.

The enrichment strategy: for every record missing a phone number, search Google Business with the company name and state. Parse the search results for phone numbers. Match against the business name to confirm it's the right company.

This runs through parallel workers. Six workers, each processing about 100 records per hour. Hit rates vary by source: some states yield 85%+ phone matches, others drop below 50%.

For large datasets, enrichment runs in the background for days. One source had 132,000 records needing phone lookup. That's a background job measured in weeks, not hours.

## Step 6: Deduplication

When you're pulling from 65+ sources, overlap is inevitable. The same entity can appear in multiple databases. Without dedup, you're inflating your numbers and degrading data quality.

The dedup pipeline:
1. Export all existing records from the production database
2. For each new import batch, check every record against existing data
3. Match on business name + state, phone number, and license number
4. Flag and skip duplicates
5. Cross-source dedup catches overlap between new batches too

One New York source had 100% overlap with existing data. Without dedup, that would have been thousands of wasted records polluting the dataset.

## What This Approach Unlocks

The contractor data was the first use case, but the same pipeline applies to any domain with public data:

**Real estate.** Property records, building permits, zoning data. All public. Scattered across county assessor websites.

**Healthcare.** Provider licensing, facility inspections, complaint records. State-level databases with the same fragmentation problem.

**Business filings.** Secretary of State records, trademark databases, corporate registrations. Every state has its own portal.

**Regulatory compliance.** Environmental permits, food safety inspections, OSHA violations. Public record, hard to aggregate.

The pattern is always the same: valuable data exists in public databases, but extracting and normalizing it at scale requires engineering effort that most people can't invest. AI-assisted development compresses that effort from months to weeks.

Public data is massively underutilized. Government databases, open data portals, and public registries contain enormous amounts of structured information. The barrier was never access. It was always the engineering effort to extract, clean, and normalize it.
